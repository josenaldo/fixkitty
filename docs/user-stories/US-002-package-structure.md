# US-002 ‚Äî Package Structure

**Epic:** Project Setup
**Phase:** 1
**Status:** Pending

## Story

**As a** developer,
**I want** the Clean Architecture package structure created under `br.com.josenaldo.fixkitty`,
**so that** layer boundaries are enforced from the start.

## Dependencies

- **US-001** ‚Äî Gradle Project Setup ‚Äî the project must exist and compile before packages can be created

## Acceptance Criteria

- All packages exist as physical directories under `src/main/java/`
- No circular imports between layers
- Package names match the design document exactly
- Every package contains a `package-info.java` with a Javadoc comment describing the layer's responsibility

## Implementation Notes

### Packages to Create

Each entry below is a package directory; create a `package-info.java` inside each one.

| Package | Layer | Responsibility |
| --- | --- | --- |
| `br.com.josenaldo.fixkitty.core.domain` | Core | Domain entities and enumerations; no dependencies on any other layer |
| `br.com.josenaldo.fixkitty.core.ports` | Core | Port interfaces that define the boundaries of the domain; no dependencies on any other layer |
| `br.com.josenaldo.fixkitty.application` | Application | Use cases that orchestrate the domain; depends only on `core.*` |
| `br.com.josenaldo.fixkitty.infrastructure.command` | Infrastructure | Implementations of `CommandRunner`; depends only on `core.*` |
| `br.com.josenaldo.fixkitty.infrastructure.privilege` | Infrastructure | Implementations of `PrivilegeManager`; depends only on `core.*` |
| `br.com.josenaldo.fixkitty.infrastructure.detectors` | Infrastructure | Implementations of `EnvironmentDetector`; depends only on `core.*` |
| `br.com.josenaldo.fixkitty.infrastructure.catalog` | Infrastructure | Implementations of `ActionCatalog`; depends only on `core.*` |
| `br.com.josenaldo.fixkitty.interfaces.gui` | Interface | JavaFX GUI controllers and components; depends on `application` and `core.*` |
| `br.com.josenaldo.fixkitty.interfaces.tui` | Interface | Lanterna TUI screens; depends on `application` and `core.*` |
| `br.com.josenaldo.fixkitty.bootstrap` | Bootstrap | Application entry point and Guice module wiring; depends on all layers |

### Key Behavior

The dependency rule for each layer is:

- `core.*` ‚Äî no dependencies on any other project layer
- `application` ‚Äî depends only on `core.*`
- `infrastructure.*` ‚Äî depends only on `core.*`
- `interfaces.*` ‚Äî depends on `application` and `core.*`
- `bootstrap` ‚Äî depends on all layers (this is the only package allowed to reference every other layer simultaneously)

Each `package-info.java` must document these rules in its Javadoc so that any developer reading the file immediately understands what the package is allowed to import.

Example for `core.domain`:

```java
/**
 * Domain entities, enumerations, and value objects for FixKitty.
 *
 * <p>This package belongs to the Core layer and must not import from any other
 * layer of the application ({@code application}, {@code infrastructure},
 * {@code interfaces}, or {@code bootstrap}).
 */
package br.com.josenaldo.fixkitty.core.domain;
```

### Additional Task

Delete the file `src/main/java/org/example/Main.java` if it exists. This placeholder was generated by the Gradle init task and must not remain in the codebase.

### Edge Cases

- If the Gradle init task created a module descriptor (`module-info.java`), evaluate whether it conflicts with Lanterna or other dependencies before keeping it; remove it if it causes unresolvable module errors
- Package names are all lowercase and follow the reverse-domain convention exactly as listed; no abbreviations or variations are acceptable
