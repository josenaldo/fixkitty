## Contexto

Primeira sessão de implementação real do FixKitty. Já existia o plano completo em `docs/plans/2026-03-23-phase1-implementation-plan.md` com 20 tarefas divididas em 6 fases (setup, domínio, casos de uso, infraestrutura, TUI, GUI).

## Abordagem de IA Usada

**Metodologia:** Subagent-Driven Development
**Skill principal:** `superpowers:subagent-driven-development`

Cada tarefa foi delegada a um subagente fresco (sem contexto acumulado), seguido de revisão em dois estágios:
1. **Revisão de spec** — verificar se o código implementa exatamente o que o plano especifica
2. **Revisão de qualidade** — verificar code quality, bugs, segurança, arquitetura

Em caso de issues Important ou Critical, um subagente de correção era despachado antes de marcar a tarefa como completa.

## Decisões Importantes

### Decisão 1: isExecutable() no enum RecoveryAction
**Contexto:** TUI e GUI precisavam distinguir ações executáveis de CHECK_ENVIRONMENT.
**Decisão:** Adicionar `isExecutable()` ao enum, em vez de comparar `action == CHECK_ENVIRONMENT` nos controllers.
**Por quê:** Mantém a decisão no domínio. Controllers não precisam conhecer a semântica de cada ação.
**Onde aparece:** RecoveryAction.java, MainMenuScreen.java, MainController.java

### Decisão 2: GuiApp.injector como public static volatile
**Contexto:** JavaFX instancia GuiApp via reflexão, impedindo injeção por construtor.
**Decisão:** Campo estático público (necessário para cross-package) + volatile (thread-safety).
**Por quê:** Pattern padrão para bridgear Guice + JavaFX. Documentado no Javadoc.

### Decisão 3: try/finally no TUI ExecutionScreen
**Contexto:** Revisão de qualidade apontou que progressWindow.close() não era garantido em caso de exceção.
**Decisão:** Envolver executeUseCase.execute() em try/finally.
**Por quê:** Evita terminal corrompido se a execução falhar.

## O Que Funcionou Bem

1. **Loop de revisão dupla** — spec review + quality review pegou issues reais antes de avançar
2. **Subagent fresco por tarefa** — sem contaminação de contexto entre tarefas distintas
3. **Testes como rede de segurança** — TDD garantiu que o bug do aggregate() fosse descoberto imediatamente

## O Que Não Funcionou / Melhorias

1. **Contexto compactado no meio da sessão** — a sessão foi tão longa que o contexto foi compactado, exigindo reconstrução do estado. Sessões futuras devem ser menores.
2. **Tasks 9-13 em um único subagent** — funcionou, mas é arriscado. Preferir um subagent por task mesmo para tarefas de infraestrutura simples.
3. **AppRestarter.getJarPath()** — identificado pela revisão final como não-funcional em dev mode. Conhecimento antecipado teria poupado uma rodada de fix.

## Bug Real Encontrado pela IA

**Onde:** `ExecuteRecoveryUseCase.aggregate()`
**O que era:** Passos com `FailurePolicy.WARN` retornavam `FAILED` em vez de `SUCCESS` quando todos os falhos eram do tipo WARN.
**Como foi encontrado:** O teste `execute_warnPolicyFailure_returnsSuccess` (adicionado durante revisão final) falhou, revelando que a ordem dos guards no método `aggregate()` estava errada.
**Fix:** Reordenar as verificações para checar `warnFailedCount == failedCount` antes de `successCount == 0`.
